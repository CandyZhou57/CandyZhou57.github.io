Nosql

nosql

1.传统架构：app—dal—mysql
2.传统架构的数据存储瓶颈：

1.数据量的总大小，一个机器内存放不下

2.数据的索引（B+Tree），一个机器内存放不下

3.访问量（读写混合）一个实例不能承受

3.数据库进化史

1.单机mysql

2.memcached（缓存）+mysql+垂直拆分——–app–dal–cache–mysql

3.mysql主从复制读写分离—–app–dal–cache–（m（write）–s（read））

4.分表分库+水平拆分+mysql集群—-app–dal–cache–（多个ms–集群）

5.mysql的扩展性瓶颈—客户–linux–nginx（负载均衡、反向代理）–tomcat集群–mysql集群–缓存集群–图片视频类资源

3.nosql–not only sql–非关系型数据库

这些类型的数据存储不需要固定的模式，无需多于操作久可横向扩展

KV、Cache、Persistance

大数据时代的3V：海量：volume；多样：variety；实时：velocity

互联网需求的3高：高并发、高可靠、高性能

4.实例分析：淘宝网站商品相关信息

1.商品基本信息–mysql/oracle

2.商品描述、详情、评价信息（多文字类的）–mongodb

3.商品的图片–其他（分布式系统文件：例如：google：GFS；淘宝：TFS）

4.商品的关键字–搜索引擎（Isearch）

5.商品的波段性的热点高频信息–内存数据库（Tair、Redis、Memcache）

6.商品的交易、价格计算、积分累计–第三方（支付宝之类）

针对数据类型多样性及多数据源等一系列问题，阿里提供了一套统一数据源

5.Nosql数据库的四大分类
  • KV键值对–redis
  • 文档型数据库（bson格式较多）–mongodb
  • 列存储数据库–（分布式文件系统）cassandra、hbase
  • 图关系数据库–（放的不是图形而是构建关系图谱，如社交网络）Neo4J、InfoGrid
6.Nosql数据库中的CAP原理CAP+BASE

1.传统的ACID

Atomicity:原子性；Consisitency：一致性；Isolation：独立性；Durability：持久性

2.Nosql的CAP

Consistency：强一致性；Availability：可用性；Partition tolerance：分区容错性

CAP理论就是说在分布式系统中，最多只能实现两点，由于网络硬件会出现延迟丢包等一些列问题，因此分区容忍性是一定要实现的，所以只能在一致性和可用性之间进行权衡

CA：传统的Oracle数据库

AP：大多数网站架构的选择（Cassandra）

CP：Redis、Mongodb

BASE就是为了解决关系数据库强一致性引起的问题而引起可用性降低而提出的解决方案

Base：基本可用（Basically Available）软状态（Soft state）最终一致（Eventually consistent）

其思想是让系统放松对某一时刻数据一致性的要求来换取系统整体伸缩性和性能上的改观。

7.redis：
  • Redis：remote dictionary server（远程字典服务）
  • 三个特点：支持数据持久化；支持数据备份（master-slave）；支持五种数据类型（String、list、set、map、zset）

Dos下redis五种数据类型的操作

String

set name “value”

get name

mset name1 “value1” name2 “value2”

mget name1 name2

del name

hash

hset hash1 key1 “value1”

hmset hash2 key2 “value2” key3 “value3”

hget hash1 key1

hmget hash2 key2 key3

hgetall hash2

list

lpush mylist aaaa //头插法（左插）

rpush mylist bbbb //尾插法（右插）

lrange mylist 0 10 //查询0到10的值

set（无序set）

sadd myset aaaa bbbb cccc

smembers myset

#如果插入信息为重复信息则第二次插入无效

Zset（有序set，按照score数值从小到大排序）

zadd Zset名称 score数值（用于排序） 添加元素
例：zadd zset 0 aaaa
zadd zset 1 bbbb
zadd zset 2 cccc
或：zadd zset 0 aaaa 1 bbbb 2 cccc // 可以同时设多个不同元素信息，其中的score数值可以重复

注：zrange zset 0 100 //查询的是按照下标从0到100含100的所有元素数据

通用操作：
del key名称 //删除key的值,可以同时删除多个Key
keys * //查询redis中所有的key名称信息
keys my* //查询以my开头的key名称信息
type key名称 //查询Key的数据类型

java8新特性

  2. StreamAPI–流–主要用来对集合进行各种便利的聚合操作或者批量数据操作

  • 流的创建

    • 集合创建流

    • 数组创建流

    • 可变参数创建流

    • 根据范围创建数值流

    • BufferReader.lines()创建流

  • 流操作（两种操作：intermediate、terminal）

    • intermediate：一个流可以后面跟随零个或多个 intermediate 操作。其目的主要是打开流，做出某种程度的数据映射/过滤，然后返回一个新的流，交给下一个操作使用。这类操作都是惰性化的（lazy），就是说，仅仅调用到这类方法，并没有真正开始流的遍历。
    • terminal：一个流只能有一个 terminal 操作，当这个操作执行后，流就被使用“光”了，无法再被操作。所以这必定是流的最后一个操作。Terminal 操作的执行，才会真正开始流的遍历，并且会生成一个结果，或者一个 side effect。
    • 除此以外，还有一种叫做short-circuiting的操作
      • 对于一个 intermediate 操作，如果它接受的是一个无限大（infinite/unbounded）的 Stream，但返回一个有限的新 Stream。
      • 对于一个 terminal 操作，如果它接受的是一个无限大的 Stream，但能在有限的时间计算出结果。
    • 常见流操作的归类
      • intermidiate：map(mapToInt、flatMap等)、filter、distinct、sorted、peek、limit、skip、parallel、sequential、unordered
      • terminal：forEach、forEachOrderd、toArray、reduce、collect、min、max、count、anyMatch、allMatch、noneMatch、findFirst、findAny、iterator
      • Short-circuiting：anyMatch、allMatch、noneMatch、findFirst、findAny、limit

  • 常见流操作详解

    • forEach：forEach就是遍历并执行某个操作，甚至对于List等实现了Collection接口的类可以不创建stream而直接使用forEach。

    • map：用来将一个集合映射为另一个集合。还有一个叫做flatMap的操作，这个操作在映射的基础上又做了一层扁平化处理。

    • filter：实现了过滤的功能

    • sorted：表示排序

    • distinct：表示去重

    • collect：在流操作中，我们往往需求是从一个List得到另一个List，而不是直接通过forEach来打印。那么这个时候就需要使用到collect了

    • 并行流：除了普通的stream之外还有parallelStream，区别比较直观，就是stream是单线程执行，parallelStream为多线程执行。parallelStream的创建及使用基本与stream类似，不过由于是并行执行，parallelStream并不保证结果顺序，同样由于这个特性，如果能使用findAny就尽量不要使用findFirst。

      使用parallelStream时需要注意的一点是，多个parallelStream之间默认使用的是同一个线程池，所以IO操作尽量不要放进parallelStream中，否则会阻塞其他parallelStream

Spring注解使用

@EnableDiscoveryClient

要想将一个微服务注册到Eureka Server（或其他服务发现组件，例如Zookeeper、Consul等），Eureka 2.0闭源之后，Consul慢慢会成为主流。须在启动类上添加注解@EnableDiscoveryClient 或@EnableEurekaClient

@valid和@validated

  • 区别：
    • 分组
    • 注解的地方：@valid可以注解在成员属性（字段）上，但是@validate不行
    • 由于@validate不能作用在成员属性上，因此不能做嵌套校验
    • @valid只能作用在controller上，@validate可以用在被其他spring管理的类上
    • @validate如果要开启方法验证，注解应该打在类上，而不是方法参数上

java中事务注解的使用

@Transactional只能被应用到public方法上,对于其它非public的方法,如果标记了@Transactional也不会报错,但方法没有事务功能.
默认情况下,一个有事务方法, 遇到RuntimeException 时会回滚 . 遇到 受检查的异常 是不会回滚 的. 要想所有异常都回滚,要加上 @Transactional( rollbackFor={Exception.class,其它异常})

Hexo常用命令

hexo d

hexo g

hexo n

1、hexo init

hexo init 命令用于初始化本地文件夹为网站的根目录

  • folder 可选参数，用以指定初始化目录的路径，若无指定则默认为当前目录

2、hexo new

  • layout 可选参数，用以指定文章类型，若无指定则默认由配置文件中的 default_layout 选项决定
  • title 必填参数，用以指定文章标题，如果参数值中含有空格，则需要使用双引号包围

3、hexo generate

  • -d 选项，指定生成后部署，与 hexo d -g 等价

详细信息请参考：https://hexo.io/docs/generating

4、hexo server

  • -p 选项，指定服务器端口，默认为 4000
  • -i 选项，指定服务器 IP 地址，默认为 0.0.0.0
  • -s 选项，静态模式 ，仅提供 public 文件夹中的文件并禁用文件监视

说明 ：运行服务器前需要安装 hexo-server 插件

详细信息请参考：https://hexo.io/docs/server.html

5、hexo deploy

  • -g 选项，指定生成后部署，与 hexo g -d 等价

说明 ：部署前需要修改 _config.yml 配置文件，下面以 git 为例进行说明

详细信息请参考：https://hexo.io/docs/deployment.html

6、hexo clean

hexo clean 命令用于清理缓存文件，是一个比较常用的命令

网站显示异常时可尝试此操作

7、Option

（1）hexo –safe

hexo --safe 表示安全模式，用于禁用加载插件和脚本

安装新插件时遇到问题可尝试此操作

（2）hexo –debug

hexo --debug 表示调试模式，用于将消息详细记录到终端和 debug.log 文件

Mockito+Junit 5

单元测试的基本原则

  • 宏观层面：AIR原则
    • automatic：全自动执行，输出结果无需人工检查，而是通过断言验证
    • Independent:为了保证单元测试稳定可靠且便于维护，单元测试用例之间决不能互相调用
    • Repeatable :单元测试通常会被放到持续集成中，每次有代码check in时单元测试都会被执行。如果单测对外部环境（网络、服务、中间件等）有依赖，容易导致持续集成机制的不可用
  • 微观层面：BCDE原则（编写单元测试代码遵守BCDE原则）
    • Border：边界值测试，包括循环边界、特殊取值、特殊时间点、数据顺序等
    • Correct：根据输入得出预期结果
    • Design：与设计文档结合，编写单测用例
    • Error： 单测的目的是为了证明程序有错。为了发现代码中潜在的错误， 我们需要在编写测试用例时有一些强制的错误输入（如非法数据、异常流程、非业务允许输入等）来得到预期的错误结果

mysql参数问题lower_case_table_names

1、参数含义：

lower_case_table_names： 此参数不可以动态修改，必须重启数据库
lower_case_table_names = 1 表名存储在磁盘是小写的，但是比较的时候是不区分大小写
lower_case_table_names=0 表名存储为给定的大小和比较是区分大小写的
lower_case_table_names=2, 表名存储为给定的大小写但是比较的时候是小写的

2、修改 参数lower_case_table_names 产生的问题

mysql> show variables like ‘lower_case_table_names’;
+————————+——-+
| Variable_name | Value |
+————————+——-+
| lower_case_table_names | 0 |
+————————+——-+
1 row in set (0.00 sec)

在mysql系统中原来的参数是lower_case_table_names=0 ，就是存储的时候区分大小写，创建了表Tt tT

mysql> create table Tt(id int);
Query OK, 0 rows affected (0.10 sec)

mysql> create table tT(id int);
Query OK, 0 rows affected (0.03 sec)

mysql> show tables;
+—————-+
| Tables_in_test |
+—————-+
| Tt |
| tT |
+—————-+
2 rows in set (0.00 sec)

修改参数 /etc/my.cnf 中设置 lower_case_table_names = 1

mysql> show variables like ‘lower_case_table_names’;
+————————+——-+
| Variable_name | Value |
+————————+——-+
| lower_case_table_names | 1 |
+————————+——-+
1 row in set (0.00 sec)

重启数据库
然后查看数据库

mysql> select * From tT;
ERROR 1146 (42S02): Table ‘test.tt’ doesn’t exist

mysql> select * From tT;
ERROR 1146 (42S02): Table ‘test.tt’ doesn’t exist

你可以发现当修改lower_case_table_names = 1后，之前创建的表名都不识别了

—创建表后，你可以发现表名不区分大小写
mysql> create table Tt(id int);
Query OK, 0 rows affected (0.23 sec)

mysql> show tables;
+————-+
| Tables_in_t |
+————-+
| tt |
+————-+
1 row in set (0.00 sec)
从此可以看出lower_case_table_names = 1，创建表名，不区分大小写

结论：不能随意修改数据库的大小写，否则表名会认不出来
lower_case_table_names 参数的修改是非动态的，必须重启数据库

3、如何修改大小写格式

如果原来所建立库及表都是对大小写敏感的，想要转换为对大小写不敏感，主要需要进行如下3步：
1.将数据库数据通过mysqldump导出;
2.在my.cnf中更改lower_case_tables_name = 1，并重启mysql数据库。
3.将导出的数据导入mysql数据库中。
